# Excel → Word Pivot

Client-side Next.js app. User pastes one or more Excel/Sheets tables; the app restructures each into an Excel-pivot-style **nested hierarchy** ("Rows area"), renders Word-ready HTML, and copies it to the clipboard for pasting into Word.

The problem this solves: wide Excel tables do not fit an 8.5x11 Word page. This app turns one wide table into a narrow, nested outline. You arrange fields into ordered **indent levels** (each level can hold one or more fields stacked at the same indentation); rows nest and merge by those levels. The **title** can **map to the destination document's Heading style** (e.g. `Heading 1`) via `mso-style-name`, so a "Use Destination Styles" paste adopts the template's heading; the **body** always uses the app's own per-level direct styling (Arial 11 default, with per-level bold).

Multiple pasted tables are managed as cards in a horizontal **tab strip** (one table edited at a time). Heading appearance is a **shared** per-level style applied to every table; each table keeps its own field choices and title. Export is per-table **Copy for Word** plus a combined **Copy all** (clipboard only — no download).

## Architecture
Pipeline (per table): paste → `parseClipboard` (Grid) → `rowsToPivotTree` (`PivotNode[]`) → `renderPivotTree` (HTML fragment) → live preview + per-table/combined "Copy for Word" clipboard write.
- `app/page.tsx` — renders the paste view
- `components/PasteInput.tsx` — parent client component; captures paste (appends a table), holds the `tables[]` array + shared per-level heading style + body font, manages the tab strip, and does combined **Copy all**
- `components/TableCard.tsx` — one table's editor: Section-title input, the per-table **Numbering** control (Off / Multilevel numbers + a **Start** input), the **Add fields** pool (unused columns), the **Structure** list (placed fields with ◄ outdent / ► indent / ▲ ▼ reorder / ✕ remove, per-field label toggles **Aa** show/hide / **B** bold / **U** underline, plus a per-field **sort** cycle ↕ off / ↑ asc / ↓ desc), the per-level **Markers** pickers, the per-level **Number level** (when numbering is on) + **Word heading** toggles, the **Blank line after** dropdown, per-card View-JSON toggle, per-card **Copy for Word**, and its `RenderedPreview`
- `components/tableModel.ts` — `TableState` type, `tableToHtml(t, titleIsHeading)` (the per-table map→render pipeline, the single source used by the card preview and combined export; threads `sortDirs` into the mapper and `breakAfter`/`numbering`/`headingLevels` into the renderer, with `titleIsHeading` setting the body heading level), and the pure bucket helpers (`addField`/`removeField`/`indentField`/`outdentField`/`moveField`/`canIndent`/`canOutdent`/`placedColumns`/`unusedColumns`)
- `components/RenderedPreview.tsx` — injects the rendered HTML via `dangerouslySetInnerHTML` (input is escaped by the renderer); scoped `<style>` styles the `ws-title` title and the pivot `[data-level]` rows. Any multilevel numbers are plain text in the fragment, so the preview shows the real numbers verbatim; a `[data-heading]` row (a level mapped to a Word heading) is shown flush-left + bold with a muted `#` placeholder (where Word inserts the live number on paste) plus a one-line hint
- `components/JsonPreview.tsx` — displays the raw parsed Grid as JSON
- `lib/types.ts` — `PivotNode`/`PivotLine`, `FieldLabel` + `DEFAULT_FIELD_LABEL`, and raw Grid types
- `lib/parser.ts` — SheetJS clipboard → Grid (`parseClipboard`)
- `lib/mapper.ts` — `cellToString` + `rowsToPivotTree(rows, levels, sortDirs?)` (ordered indent buckets → `PivotNode[]`; each bucket is one level holding ≥1 field as `PivotLine{col,name,value}`, merged by a composite key) + a pure post-pass `sortTree` that reorders sibling groups by `sortDirs` (per-level, the bucket's first sort-enabled field; `localeCompare` with `numeric:true, sensitivity:"base"`, reversed for `desc`; stable, so no dir = first-seen order)
- `lib/renderers.ts` — `renderPivotTree(nodes, title?, markers?, fieldLabels?, breakAfter?, numbering?, headingLevels?, titleIsHeading?)` (optional title row; each node's `lines` → `<p class="ws-lvl" data-level="N">` with the label `name:` shown/bolded/underlined per `fieldLabels[col]` then the value; marker on the first line only; when `numbering.mode==="multilevel"` the first line is prefixed with an app-drawn compounded number (`5.1`/`5.1.1`, from `start`) as plain escaped text and the marker is suppressed; a `headingLevels[depth-1]` level's first line gets `data-heading="K"` (K = `(titleIsHeading?1:0)+depth`) and its number/marker is suppressed (Word numbers it) while the path still computes for children; after a node's whole subtree, `breakAfter[depth-1]` pushes an nbsp spacer `<p>`; `data-level` clamped at 9); `NumberingConfig` + `DEFAULT_NUMBERING`, `MarkerKind`/`markerText`/`defaultMarker`; escapes user text. Private `escapeHtml`/`wrapLabel`/`multilevelNumbers` + `toAlpha`/`toRoman`
- `lib/clipboard.ts` — `HeadingStyle` (`{ levels, indentStep, headingStyleName }`)/`LevelStyle` types; `isHeadingStyleSet(name)` (whether a Heading style is set — sets the body heading level); `buildWordHtml` (Word doc wrapper: the title → `<p class="MsoTitle">` + a `mso-style-name:"<heading>"`+`mso-outline-level:1` rule when `headingStyleName` is set (else inline level-1 look); each `data-heading="K"` row → `<p class="MsoHeadingK">` + a `mso-style-name:"Heading K"`+`mso-outline-level:K` rule (so it enters the Word outline — nav pane + collapsible); every other body row (including nbsp spacers and app-drawn multilevel numbers, already plain text) → a `<p>` with **inline** direct formatting (per-level color/font/size + indent + compact spacing) and `<b>`/`<i>`/`<u>` label runs — inline so it survives a "Use Destination Styles" paste; `sanitizeStyleName` (allow-list)) + `htmlToPlainText`
- `docs/` — OVERVIEW, ARCHITECTURE, ROADMAP

## Data Model
The product is a recursive pivot tree, not a flat table:
- `PivotNode { lines: PivotLine[], children: PivotNode[] }`; `PivotLine { col, name, value }` keeps each field's label `name` (header) and `value` SEPARATE + raw (the renderer escapes/formats them). A node's `lines` are the fields at one indent level (≥1), rendered stacked; the first line carries the level's marker (or, when multilevel numbering is on, its app-drawn static number, which replaces the marker). Rows whose values match across all of a level's fields **merge** (composite group key).
- The pivot structure is `pivotLevels: number[][]` — ordered **indent buckets**; bucket index = indent depth, each bucket a non-empty ordered list of grid columns. Columns in no bucket are unused (hidden).
- `FieldLabel { show, bold, italic, underline }` (per grid column, in `TableState.fieldLabels`) controls each field's label prefix (`name: `): hide it, or bold/italic/underline it. Bold and italic cover the whole `name: `; underline covers only the name (not the `: ` separator). Default = shown, plain.
- `sortDirs: Record<col, "asc"|"desc">` (per grid column, like `fieldLabels`) sorts the sibling groups at that field's level; absent col = off (first-seen). `breakAfter: boolean[]` (per indent level, like `markers`) toggles a blank spacer line after that level's groups. `numbering: { mode: "off"|"multilevel"; start: string; levels: boolean[] }` (per table) drives app-drawn static multilevel numbers — `start` is a dotted-decimal string (the exact number of the first top-level item, e.g. `"5.1"`), `levels` shows/hides the number per indent level (`NumberingConfig` from `lib/renderers.ts`). `headingLevels: boolean[]` (per indent level) maps a level's rows to a destination **Word heading** (nav pane + collapsible, flush-left); Word supplies their number, so the app number/marker is suppressed on them.
- Rendered as a title row (`<p class="ws-title">`) + nested rows (`<p class="ws-lvl" data-level="N">`, a `data-heading="K"` attr on rows the user mapped to a Word heading); `buildWordHtml` maps the title AND any heading-marked levels to Word heading styles, and the rest of the body (markers, app-drawn numbers, spacers) to the app's direct per-level look.

## Pivot view
The one view (`rowsToPivotTree`) replicates Excel's "Rows area" as ordered indent buckets:
- **Add fields** — a pool of the columns not yet placed; clicking one appends it as a new deepest single-field level.
- **Structure** — the placed fields as an indented list. Per field: ◄ **outdent** / ► **indent** (set the indent level; stacking fields at one level shows them together and merges by the composite of that level's values), ▲/▼ **reorder** (swap with the flat-order neighbour, keeping the bucket shape), ✕ **remove**, four label toggles — **Aa** (show/hide the `name:` label prefix → "Item Category: Fruit" vs "Fruit"), **B** (bold the label — also bolds that line's leading number/marker), **I** (italicize the label), **U** (underline the label — only the name, not the `: ` separator) — and a **sort** cycle (↕ off / ↑ ascending / ↓ descending) that orders the sibling groups at that field's level. Emphasis targets the label only; the value stays plain. Label look (`fieldLabels`) and sort (`sortDirs`) are both stored per grid column, so they persist across remove/re-add.
- **Sorting** (per field, `sortDirs: Record<col, "asc"|"desc">`) — a post-pass on the built tree reorders the SIBLING groups at a field's indent level by that field's value: one comparator (`localeCompare` with `numeric:true, sensitivity:"base"`) auto-handles numbers and text ("2" before "10", text alphabetical + case-insensitive), reversed for `desc`. Off keeps first-seen order; a stable sort keeps equal-keyed siblings in first-seen order. When a bucket stacks several sorted fields, the first in bucket order wins.
- **Markers** (per-table, `markers: MarkerKind[]` indexed by indent level) — a `<select>` per level picks its marker style: `1.` / `1)` / `A.` / `a.` / `I.` / `i.` / `•` / `–` / None. Sparse → `defaultMarker(depth)` (the `1./a./i.` cycle). Only the FIRST field of a multi-field level is marked; the title is never marked, and a numbered node's marker is suppressed (the number replaces it).
- **Blank line after each** (per-table, `breakAfter: boolean[]`) — a dropdown picks ONE level (labelled by that level's field name, e.g. "MISSION RULE TITLE", or none); the renderer then pushes one empty spacer paragraph (nbsp, no marker/number/label) after each of that level's groups' whole subtree, so the preview and the Word paste show a blank line before the next group at that level (after each title → a gap before the next title). Sibling counting is by node position and spacers are emitted after the subtree, so they never perturb the marker/number counters.
- **Numbering** (per-table, `numbering: { mode, start, levels }`) — app-drawn STATIC multilevel numbers (NOT Word auto-numbering). When `multilevel`, the renderer (via `multilevelNumbers`) prefixes each node's first line with its number as plain body text. `start` is a dotted-decimal STRING that is the EXACT number of the first top-level item: the top shown level reads it with its LAST component advanced per sibling (`Start "5.1"` → 5.1, 5.2 across top items) and each deeper shown level appends `.<1-based index>` (5.1 → 5.1.1 → 5.1.1.1). So to nest a body under a `5.0` **Section title**, set `Start "5.1"`. The number replaces the marker on that node, and the leading number/marker inherits the field's bold. A per-level **Number level** checkbox (`numbering.levels`, indexed by indent level) shows/hides each level's number; a hidden level is **transparent** — its line goes plain AND its children CONTINUE the nearest shown level's sequence, so the numbers form a contiguous outline of only the shown levels (number TYPE + TEXT, hide TITLE → `5.1` then `5.1.1`, `5.1.2`… — it *follows* the gap, never a number under a missing parent, and the shared counter across hidden siblings means no collisions). `start` is a free dotted decimal (digits + dots); numeric only for now (the marker code is structured so letter/roman styles could be added later). The numbers themselves never make a row a Word heading (see **Word heading** below for that); the preview shows the real numbers verbatim.
- **Word heading** (per-table, `headingLevels: boolean[]`, per indent level, labelled by field name) — map a level's rows to a destination `Heading K` style (K = `(titleIsHeading?1:0)+depth`, so under a `Heading 1` title the top body level → `Heading 2`). On a **Use Destination Styles** paste those rows enter the document outline — **Navigation pane + collapsible**, flush-left (they lose the body indent) — and **Word supplies their number**, so the app's number/marker is suppressed on them. A heading level still establishes a numbered BASE, so deeper app-numbered rows nest AND **reset under each heading group**: TYPE → Word `5.1`/`5.2`, its TITLEs → app `5.1.1`/`5.1.2`, then `5.2.1` under the next type. Set **Start** to the heading's section number (e.g. `5.1`) so the app's deeper numbers line up with Word's heading number. Use it on just the top level or two, so the nav gets a few useful entries (not a flood). The preview shows those rows flush-left + bold with a `#` placeholder for Word's number.
- **Section title** (optional) — the title row (`MsoTitle`). Maps to the **Heading style** name when set (a real Word heading, `mso-outline-level:1` → "5.0", one Nav-pane entry per table); else a plain level-1 row. With a title the nested data starts at level 2; without one, level 1. Numbering is independent of the title (top data siblings number from `start` whether or not a title exists; the title is never numbered).

## Styling
**Title → Word heading mapping:** the **Heading style** input (default `Heading 1`) maps ONLY the title via `mso-style-name`+`mso-outline-level:1`. On a **Use Destination Styles** paste the title adopts the destination document's heading style (font/size/color/numbering from the template); blank = the app's direct level-1 look. That is one Nav-pane entry per table. **Body rows are mapped only when you mark a level as a Word heading** (then `data-heading="K"` → `Heading K`, so those rows enter the nav pane + collapse); every other body row (markers, app-drawn multilevel numbers, spacers) uses the app's per-level look, emitted inline (so it survives the paste and matches the live preview), with per-field label `<b>`/`<i>`/`<u>` runs on top. App-drawn multilevel numbers are plain body text, so an un-marked body never becomes a Word heading and Word's Nav pane / TOC stay clean.

One **shared per-level** styling panel ("Level styles") drives the look across all tables — `HeadingStyle = { levels: LevelStyle[]; indentStep: number; headingStyleName }`. Each level row sets color/font/size/bold; **defaults are plain Arial 11 black** (to match a document body), with a "Reset levels" button. **Level 1** styles the title row (when not mapped); **Levels 2-9** style the nested rows by depth. The panel shows one row per depth in use. Beside it: a **Body font** control (document body font) and an **Indent/level (in)** control (`indentStep`, clamped 0–2; nested rows indent `(n-1)×step`). Body paragraphs use compact spacing (`line-height:115%`, no space before/after). All style inputs are form-controlled (hex color, allow-listed font, clamped numbers), so no free user text lands in a `style`/`class` attribute.
